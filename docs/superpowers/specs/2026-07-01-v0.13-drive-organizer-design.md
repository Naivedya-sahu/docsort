# v0.13.x — Drive Organizer: Dedup, Tagging, Search, KB Integration

Status: brainstormed, approved section-by-section, pending written-spec review.
GUI/graph-view work (Explorer tab, Graph tab, visual overhaul) is explicitly deferred to v0.14.x — this
version is backend/engine-only; new capabilities surface through the **existing** Flet GUI (Reports tab
extended with new report types), not new nav items.

---

## 1. Context & motivation

docsort today classifies and renames files one at a time inside a single folder pass. Real usage
surfaced two problems that the current engine can't address:

1. **Processing time at scale.** Every file costs at least one local-LLM call (TEXT tier minimum,
   escalating to VISION/FRONTIER for hard cases). At thousands of files this is too slow given
   hardware constraints (4GB VRAM laptop) and the user's <4-5s/file expectation at scale.
2. **The real blocker isn't tagging, it's mess.** The user wants to migrate a personal archive into a
   Second-Brain Obsidian KB, but duplication and unmanageable folder depth block that migration. A test
   tree export (`Drive_D.txt`, 5304 entries) was analyzed and confirmed this concretely:
   - Max folder depth 11, bulk of files sitting 7-10 levels deep.
   - A folder (`Remarkable 2`) was duplicated **wholesale** in two other locations — 695/695 files
     identical in one copy, another 20/20 files a full subset in a third. This kind of duplication is
     invisible to per-folder dedup (the user's existing hygiene rule scopes DupeGuru to a single folder,
     never across roots — which is exactly why these survived).
   - ~1334 files (eagle_libraries-master, an `iir1` C++ library's generated docs) were vendored
     third-party content, not personal material, padding both file count and depth.
   - At the user's actual full-drive scale, most future duplicates won't be tidy exact-copies or clean
     duplicate subtrees — they'll be scattered near-duplicates (re-exported, re-scanned, renamed) with no
     matching structure to compare against ("dupe hell").

This version's job: build the engine capability to detect and clean this mess, tag what remains fast
enough to be usable at scale, and produce output that plugs into the already-designed Second-Brain
integration contract.

## 3. Architecture

### 3.1 Ground-truth index (`docsort/index.py`, new)

SQLite table: `path, size, hash, embedding, tags, dupe_group, subtree_signature, archive_depth,
source_archive, mtime`. This is a **queryable snapshot of current drive state** — distinct from the
existing `_docsort_state.jsonl` journal, which stays exactly as-is (append-only *action log* powering
`--undo`/`--resume`). The index is rebuildable/disposable; the journal is the durable audit trail.
Scoped **cross-root**, not per-folder — this is the specific fix for the duplicate-subtree blind spot
found in the test data.

### 3.2 Archive-aware scanning (extends the index scanner)

`.zip` contents are walked recursively via Python's stdlib `zipfile` (no external dependency), reading
entry bytes directly for hashing/embedding without full extraction to disk. Each internal file becomes
its own index entry with a virtual path (`outer.zip → inner/path/file.pdf`) and an `archive_depth` field.
Nested zips (zip-under-zip) recurse the same way, bounded by:
- **Max nesting depth** (default 6) — exceeding this is *reported*, not silently skipped.
- **Max extracted-size budget per archive** — zip-bomb guard.

Once archive-internal files are index entries, every downstream layer (dedup, classify, reorg) treats
them like any other file — no separate code path needed. This directly targets "zips under zips" and
"copies within copies" — a zip 100%-redundant with already-organized loose files becomes a visible,
reviewable delete-candidate instead of an opaque blob nobody looks inside.

### 3.3 Everything integration (`docsort/everything.py`, new)

Wraps Everything's local HTTP query API for fast full-drive/folder enumeration, replacing/augmenting a
recursive `os.walk` for the scan phase. Also backs ad-hoc search (`docsort --search "term"` or by tag)
without requiring a fresh scan.

### 3.4 Classification cascade

**New EMBED tier**, inserted before the existing TEXT tier: cheap CPU embedding (filename + path +
first-page text) compared via cosine similarity against per-label centroid embeddings (built once from a
handful of example files per STREAM/SUBJECT — zero-shot, no training step). Clears a confidence
threshold → tag directly, **zero LLM calls**. Below threshold → falls through to the existing
TEXT→ESCALATE→VISION→FRONTIER pipeline, unchanged. This is the standard cascade/routing pattern from
cost-aware LLM serving research (RouteLLM, FrugalGPT-style): cheap tier resolves the bulk, expensive
tiers only see the genuinely ambiguous tail. Directly addresses the "ages for thousands of files"
problem without touching classification quality on hard cases.

**Hybrid tag output**, same call — no added latency: TEXT/EMBED tiers keep the current single-line
STREAM+SUBJECT reply; VISION/FRONTIER tiers' *existing* single inference call is asked for a richer
schema (multiple STREAM/SUBJECT pairs) instead of a second call. Parser in `runcore.py`/`cli.py` extended
accordingly. Rationale: multi-tag value is highest exactly where confidence is lowest (VISION/FRONTIER),
and this must not compound the cost problem those tiers already have.

### 3.5 Clean phase (`docsort/dedup.py`, `docsort/vendor.py`, new)

Three dedup layers, all index-wide (cross-root), all **review-only — nothing is ever auto-deleted**:

1. **Exact-hash** — wraps DupeGuru's hashing rather than reimplementing it; byte-identical files.
2. **Subtree-signature** — structural signature (sorted relative filenames + hashes) per folder,
   catches whole duplicated folder trees (the `Remarkable 2` ×3 case). An archive's internal listing is
   just another subtree signature — no special-casing needed.
3. **Near-duplicate embedding (new, added after real-data review)** — reuses the EMBED-tier vectors
   already computed for classification (zero extra compute) for global cosine-similarity clustering
   across the whole index. Catches scattered content-duplicates with no matching folder structure — the
   specific "dupe hell at scale" scenario exact-hash and subtree-signature both miss. Highest
   false-positive risk of the three, so it is presented as its own clearly-labeled review category, never
   auto-merged.

**Vendor-dump detector** — heuristic match on known third-party project markers (e.g. a project name
matching a known OSS library, generated-docs directory shape like Makefile+Doxygen output) — flags
delete-candidates, human confirms, same as the dedup layers.

All three dedup layers + vendor detection land in one combined review report (extends the existing
Reports tab); confirmed deletions apply via `--apply-clean`, journal-backed and undoable like every other
docsort mutation.

### 3.6 Reorg-suggester (`docsort/reorg.py`, new)

Runs after Classify (needs tags/embeddings to cluster meaningfully) and after Clean (no point clustering
duplicate or vendor junk). Reuses EMBED-tier embeddings for agglomerative hierarchical clustering
(same family of technique as the `messy-folder-reorganizer-ai` / HERCULES approaches found in research) to
propose a flatter target structure. **One naming call per cluster, not per file** — same cost-control
principle as hybrid tagging. Dry-run move-list, human approves each move, `--apply-reorg`, same
journal/undo pattern as file renames today.

### 3.7 Second-Brain integration contract

The Second-Brain KB (separate repo, `naivedya-sahu/second-brain`, path
`D:\Vault\Personal\Active\Second-Brain\Repo\Second-Brain`) already designs this integration as its own
"Phase 2," documented in their `docs/PROJECT_CONTEXT.md` §7. Key points already confirmed compatible,
**no docsort code changes required for the contract itself**:
- Their 13-domain KB schema (`00`–`13`) maps 1:1 to docsort's SUBJECT codes — verified, no drift.
- Their reader needs the log CSV columns docsort already writes:
  `path,old,new,stream,subject,type,conf,source`.
- **Files stay where docsort puts them; the KB stores links (via `## Sources`), never copies.** docsort
  does not push output into their `raw/inbox/` — that inbox is for unrelated new content.
- Their doc has stale naming (calls this repo `doc-handler`, log file `_doc_handler_log.csv`) — pre-rename
  names; a documentation fix on their side, not a docsort change.

**Still open, belongs to the user/Second-Brain side, not this design:**
- STREAM `REC` (records/admin) has no KB-area mapping yet in their STREAM→area table.
- Whether docsort's 13 SUBJECT codes become 13 separate `10 Knowledge/<Subject>/` folders or collapse
  under broader categories — their own folder-growth rule (folder only at ~5 children) means either is
  safe and reversible, defaulting to 13 separate (preserves existing alignment) is the lower-risk choice.

### 3.8 External execution toolchain (File Juggler + DupeGuru + empty-sweep + WinDirStat)

docsort is the **decision engine** (tags, dedup/vendor detection, quarantine staging) — it does not need
to own every physical move/delete/review UI itself. The user's actual day-to-day execution splits across
four already-trusted external tools, and docsort's output is shaped to feed them directly, no adapter
needed:

- **File Juggler** (rule-based file mover) — watches for the `[STREAM-SUBJECT]` filename prefix docsort
  already produces and physically routes files into the target tree per the user's own File Juggler
  rules. This becomes the **ground-truth sorted folder**. docsort's own `--apply`/`--move` stays as the
  underlying/scriptable path (same journal-backed result either way) — File Juggler is the hands-on front
  end for this user, not a replacement engine.
- **Quarantine folder** — Clean-phase delete-candidates (dedup + vendor-dump) are moved (existing `--move`
  mechanism, pointed at a holding directory instead of the sorted tree) out of the working tree, so
  File Juggler's rules don't file them and DupeGuru has a clean "normal" side to scan.
- **DupeGuru, reference mode** — Reference = the ground-truth sorted folder, Normal = the quarantine
  folder. DupeGuru's own comparison UI finds the matches; human manually reviews and sends true dupes to
  Recycle Bin. docsort's Clean-phase (subtree-signature + near-dup embedding) is what made this possible
  in the first place — DupeGuru's reference mode only compares folders you point it at, it doesn't
  discover cross-root or scattered duplication on its own; docsort's job is finding *where* to point it.
- **Audit loop** — whatever survives DupeGuru in the quarantine folder wasn't actually junk. Re-run
  Classify over the survivors to file them properly instead of manually reviewing stragglers one by one.
- **Empty-folder / 0KB-file sweep** (user's existing tool) — after File Juggler and DupeGuru have moved
  and recycled real content, the original source locations are left full of now-empty folders and
  zero-byte stragglers. This tool sweeps those out before Reorg-suggest runs, so clustering isn't working
  against a tree full of dead directory noise.
- **WinDirStat** (external, review-only) — final visual pass over the outcome. This is the same
  treemap-style review docsort's own v0.14.x Explorer tab will eventually provide in-app; until that
  ships, the real WinDirStat is the intended final check, no need to wait for v0.14.x to get it.

### 3.9 UI changes (v0.13.x, additive only)

No new nav items — Explorer/Graph stay v0.14.x. Everything else needed is additive to the **existing**
Flet tabs:
- **Run tab** — add an EMBED-tier hit-rate counter (e.g. "812/900 resolved at EMBED, 88 escalated") to
  the existing counters row — the visible proof the cascade fix is working.
- **Reports tab** — two new report types: **Clean report** (dupe groups + subtree-dup pairs + near-dup
  clusters + vendor-flagged list; archive-internal paths render as `outer.zip → inner/path`) and **Reorg
  proposal** (move-list). Each gets an apply button, same visual pattern as the existing v0.12.0
  "Apply audited" button — `Apply Clean` (moves confirmed items to quarantine), `Apply Reorg`.
- **Tags tab** — split view: two-axis entries (EMBED/TEXT tier) vs multi-tag entries (VISION/FRONTIER
  tier), so the hybrid tag model doesn't muddy the current single STREAM/SUBJECT editor.

**Search stays CLI-only this version** (`docsort --search "term"` / `--search --subject 04BJT`). Everything
integration is an internal Clean-phase discovery mechanism (cross-root dedup, "diversity of a file in the
whole filesystem"), not a workflow-required interactive feature — a GUI results-list would overlap with
the Explorer tree work already deferred to v0.14.x, so building it twice isn't justified.

**New thresholds/knobs are config-file entries, not new GUI controls** (EMBED confidence threshold,
near-dup similarity threshold, archive max-nesting-depth, archive max-extracted-size, vendor-dump marker
list) — same `config.py`/per-user config file pattern docsort already uses for DEFAULTS, not a new slider
or toggle per option. Reduces GUI surface for power-user knobs that don't need one-click access the way
Run/Stop/Apply do.

## 4. User workflow

Same dry-run → review → apply pattern docsort already uses everywhere; no new interaction model to learn.

1. **Scan** — point docsort at a root (folder picker/path, same as today). Everything API (or `os.walk`
   fallback) plus the archive-aware walker builds/refreshes the index. Cheap, no LLM, re-runnable anytime.
2. **Clean** — 3-layer dedup + vendor-dump detection, one combined review report (existing Reports tab).
   Confirmed delete-candidates move to a **quarantine folder** via `--apply-clean` (journal-backed,
   undoable) rather than being deleted directly.
3. **Classify** — existing `docsort` run, tier chain now `EMBED → TEXT → ESCALATE → VISION → FRONTIER`.
   Output format unchanged (`[STREAM-SUBJECT]` prefix or multi-tag variant + log row) — Second-Brain's
   reader needs no changes.
4. **File Juggler** (external, user's own rules) routes the renamed files into the target tree, forming
   the ground-truth sorted folder — see §3.8.
5. **DupeGuru, reference mode** (external) — reference = ground-truth sorted folder, normal = quarantine
   folder. Manual review, Recycle Bin. See §3.8.
6. **Audit the remainings** — re-run Classify over whatever DupeGuru didn't flag as a dupe; it's unique
   content that needs filing, not further cleanup.
7. **Empty-folder / 0KB-file sweep** (external, user's tool) — clears the folder stragglers File Juggler
   and DupeGuru left behind before Reorg-suggest runs — see §3.8.
8. **Reorg-suggest** — proposes flatter structure from Classify's tags/embeddings; dry-run move-list,
   `--apply-reorg` on confirmation.
9. **WinDirStat** (external, review-only) — final visual check of the whole operation's outcome —
   see §3.8.
10. **Search, anytime, orthogonal to the phases** — Everything-backed, by name or by tag, no fresh scan
    required.
11. **Handoff (not a docsort step)** — once files are clean/tagged/deduped in place, Second-Brain's own
    `ingest.py --from-doc-handler` (their repo) walks the result and creates KB note links. docsort's job
    ends at producing a stable, correctly-tagged, deduped tree.

## 5. Worked example (real data, `Drive_D.txt`)

Measured (not projected) against the 5304-entry test tree:

| Stage | Files | Dirs | Max depth |
|---|---|---|---|
| Before | 4893 | 411 | 11 |
| After Clean (dedup + vendor removal) | 2864 (-41%) | 290 (-29%) | 11 (unchanged — remaining depth is genuine unique content, Reorg's job) |

Findings that shaped this design:
- `Remarkable 2` (top-level) vs `Sorting/Surface/RM/Remarkable 2` (nested): 695/695 files identical —
  full duplicate tree, invisible to per-folder dedup.
- `Remarkable 21`: 20/20 files fully subset of `Remarkable 2` — fully superseded.
- Vendor dump (`eagle_libraries-master` + `iir1` generated docs): 1334 files / 50 dirs.
- Classify-phase illustration: files in the *same* source folder (`Academics/Section_3`) correctly split
  across STREAMs by content, not folder — `Network_Analysis/*.pdf` → `CW-01CA`, but
  `MY GATE NOTES (New).pdf` in the same folder → `GATE-01CA`. Per-file classification catches this;
  folder-only reorganization would not.

## 6. Safety & error handling

Every mutating operation (`--apply-clean`, `--apply-reorg`, plus existing `--apply`/`--apply-journal`)
follows the pattern already established in docsort: dry-run by default, explicit human confirmation to
apply, journal-backed so `--undo` reverses it. Near-duplicate detection (highest false-positive layer)
is never auto-applied under any flag — always a labeled review category. Archive nesting-depth overflow
is surfaced in the report, never silently truncated.

## 7. Testing approach

Consistent with existing `tests/test_core.py` / `tests/test_runcore.py` — pure-logic, hermetic, no
model/network dependency:
- Index/scan logic: unit-test against a synthetic file tree (including nested zip fixtures), no real
  Everything/LM Studio dependency.
- Dedup layers: unit-test exact-hash and subtree-signature against fixture trees with known duplicate
  patterns (including the archive-nested case). Near-dup embedding logic tested with pre-computed fixture
  vectors (no live embedding model call in CI).
- Cascade tier routing: unit-test the EMBED→TEXT escalation decision against fixture
  confidence-score inputs, not a live model.
- GUI additions (new report types in Reports tab): validated by headless control-tree construction per
  existing GUI-testing convention; human visual smoke-test remains the merge gate.
